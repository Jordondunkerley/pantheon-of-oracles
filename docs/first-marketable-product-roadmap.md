# Pantheon of Oracles – First Marketable Product Roadmap

## Source rule
This roadmap is grounded in the Pantheon of Oracles backend files and patch history. Where patches conflict, newer patch logic should take precedence. This roadmap is intentionally scoped to the **first marketable core product**, not the full long-term franchise.

## Product identity
**Product name:** Pantheon of Oracles
**Role in franchise:** Canonical oracle source engine and first marketable desktop product.
**Parallel product rule:** Future products (including Pantheon of Oracles: Clash) should consume persistent oracle identities from the core system rather than fork disconnected oracle versions.

---

## What the first marketable product must prove
The first product is worth selling when a user can:
1. create or load a player profile
2. generate their astrology profile directly from birth data (with import as a secondary path, not the primary requirement)
3. awaken a meaningful oracle council
4. configure a preferred AI provider/model
5. interact with distinct oracle personalities in a persistent way
6. understand that these same oracles can later persist into future Pantheon products

This is the minimum commercial proof loop.

---

## What belongs in scope for the first marketable product
### Core in-scope systems
- Player account/profile layer
- Astrology chart generation from birth data + structured display
- Oracle profile registry
- Oracle awakening / creation flow
- Oracle interaction workspace
- Provider/model configuration
- Canonical oracle import/export path
- Product-quality onboarding and demo flow
- Voice/audio readiness architecture

### Important but secondary for this phase
- Guild/host/friend oracle metadata hooks
- Faction and planetary alignment framing
- Codex/Oracron framing as future continuity layer
- Light monetization framing (founder / access-tier awareness)

### Out of scope for first saleable product (for now)
- Full multiplayer gameplay execution
- Full raid / dungeon / warband loop
- Descendant council systems
- Throne worlds / leviathans / fusion
- Full video embodiment
- Full Clash implementation

These should remain future hooks and roadmap signals, not blockers.

## Product milestones

### Milestone 1 – Canonical Core (in progress)
Goal: establish the desktop app as the source of truth for oracle identity.

Must include:
- stable oracle schemas
- astrology profile layer
- player profile layer
- provider config layer
- oracle import/export groundwork

Status:
- substantially underway

### Milestone 2 – Usable Oracle Experience
Goal: make the product feel genuinely usable rather than only well-structured.

Must include:
- polished onboarding
- believable seeded user journey
- persistent session workspace
- distinct oracle voices
- clearer provider/session binding

Status:
- underway

### Milestone 3 – Saleable Prototype Layer
Goal: prepare the product to be shown, explained, and sold as founder / prototype access.

Must include:
- cleaner release docs
- demo script
- market positioning
- Windows handoff path
- stronger polish and coherence

Status:
- underway / partial

### Milestone 4 – Audio Step
Goal: make oracle communication feel significantly more alive.

Must include:
- voice profile metadata normalized across canonical oracles
- audio output path design
- oracle session model that can support text + voice continuity

Status:
- early groundwork only

---

## Highest-priority next builds
### 1. Canonical schema normalization
Need to normalize all seeded oracles to the same richer template shape, including:
- preferred voice profile
- audio readiness
- avatar readiness
- clearer franchise traits
- stronger combat/export descriptors

Why this is next:
Because the core product is now becoming a source engine, and inconsistency in oracle records weakens everything downstream.

### 2. Canonical oracle package improvement
Need to improve exported package quality for future products by tightening:
- combat style descriptors
- weapon signatures
- visual silhouette descriptors
- spirit identity / creature metadata
- product compatibility flags

Why this matters:
This is the bridge to future products like Clash.

### 3. Session realism pass
Need to improve the oracle interaction workspace so it feels closer to a true product experience.
Potential work:
- richer session metadata
- stronger demo conversations
- future provider-response insertion point
- better oracle chamber framing

### 4. Audio-ready UX pass
Need to start exposing the future audio path in a way that makes the product feel like it is moving somewhere compelling.

---

## Recommended work order from here
1. Canonical schema normalization
2. Canonical package/export improvement
3. Session realism pass
4. Audio-ready UX pass
5. Desktop packaging/polish loop
6. Earliest saleable founder/prototype release framing

---

## Selling threshold
The first product becomes worth selling when:
- it feels coherent
- the core loop works
- oracle identity feels distinct and persistent
- import/export story is credible
- users can imagine their pantheon persisting into future products

That threshold is lower than full completion. It does not require the whole game universe to be finished.
