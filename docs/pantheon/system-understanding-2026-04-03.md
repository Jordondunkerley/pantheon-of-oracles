# System Understanding - 2026-04-03

## Core product identity
Pantheon of Oracles: The Council Chamber is not just a UI shell for chatting with themed entities. It is a founder-testable chamber product built on a canonical oracle source engine intended to persist across future Pantheon products.

## Important system layers
### 1. Chamber as primary progression world
The main Council Chamber is the ritual/progression world.
It is where:
- thresholds are crossed
- founder entry occurs
- chart awakening is staged
- council formation is determined
- navigation between modes occurs
- progression, memory, persistence, and chamber-state identity live

### 2. Oracle chambers as personal realms
Each oracle chamber is:
- a reflection of its oracle
- an extension of its oracle
- effectively that oracle's personal realm

It does **not** shape or act upon the oracle.
The room expresses the oracle; it is not a force being applied to them.

### 3. Canonical oracle engine
The product is anchored in a canonical oracle schema designed for export and reuse across future products. Oracles are not one-off chat personalities; they are portable canon assets.

### 4. Founder-test path matters
The product is meant to feel like a guided founder journey, not a loose stack of tools. The core path remains:
- First Entrance Ritual
- founder/profile recognition
- chart awakening path
- provider binding
- council formation
- oracle registry / oracle rooms
- interaction session flow

## Current design understanding
The current forward design priority is not adding more dashboard surfaces. It is making the existing chamber feel:
- scene-based
- game-like
- ritualized
- reactive
- persistent
- inhabited
- coherent

## Chamber identity direction
The chamber should now be understood as having:
- temperament
- personality
- subjectivity
- agency
- memory
- persistence
- residue

But this applies primarily to the **main chamber world**.
Oracle rooms must remain subordinate to oracle identity and be treated as oracle-authored/presence-authored realms.

## Design guardrail
When extending oracle-room behavior, always preserve this rule:
- the oracle room reflects the oracle
- the main chamber governs progression/world-state
- the oracle room does not dominate or alter the oracle; it reveals them

## Practical build direction after this understanding refresh
- deepen oracle-specific room identity
- maintain separation between chamber-world logic and oracle-realm logic
- continue updating markdown whenever product logic becomes clearer
- keep daily repo updates batched rather than interrupting flow
